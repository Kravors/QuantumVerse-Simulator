# QuantumVerse Simulator – Comprehensive Functional Verification Master Plan

## 1. Executive Summary

This plan provides a systematic methodology to verify the correctness, robustness, and performance of every public function, class, and module within the QuantumVerse Simulator. It builds upon the existing 51‑test foundation, expands coverage to all exposed APIs, and introduces integration, UI, performance, and acceptance tests. The goal is to guarantee that every advertised capability behaves as documented under both normal and edge‑case conditions.

## 2. Scope

The verification covers all modules listed in the API Reference:

- **Spacetime** (Event4D, MetricTensor, LightCone)
- **Physics** (GeodesicIntegrator, DifferentiableGeodesicIntegrator)
- **UI4D / ImGui** (UI4D_ImGui, FloatingPanels, TimelineBar, ContextMenu, etc.)
- **Discovery** (DiscoveryEngine, TheoryPlugin, 10+ instruments)
- **Quantum Gravity** (CausalSetEngine, CDTEngine, SpinFoamEngine, GFTEngine)
- **Math Utilities** (Vector4D, Matrix4x4, SO4Rotation, AutoDiff)
- **Rendering** (CurvatureRenderer, QuantumGeometryRenderer, CelestialBodyRenderer)
- **VR** (MultiUserServer)
- **ML** (GeodesicNeuralODE, CurvatureNormalizingFlow)

For each public method, we define test objectives, input variants, expected results, and the level (unit, integration, UI, performance).

## 3. Testing Hierarchy & Strategy

| Level | Description | Tools / Approach |
|-------|-------------|------------------|
| **Unit** | Isolated function/class tests | Google Test (existing framework), custom test harness |
| **Integration** | Interaction of two or more modules (e.g., geodesic + metric) | Google Test with linked libraries |
| **UI** | ImGui panels, input handling, rendering | Automated screenshot comparison (e.g., OpenCV) + manual exploratory |
| **Performance** | Timing, memory footprint, scalability | Benchmark tests, `std::chrono`, Valgrind/AddressSanitizer |
| **Acceptance** | End‑to‑end scenario tests | Pre‑defined workflows, build & deploy scripts |
| **Stress / Edge** | Extreme values, large data, long runs | Parameterised tests, fuzzing |

## 4. Module‑by‑Module Test Plan

### 4.1 Spacetime Module

#### 4.1.1 Event4D

| Function | Test Type | Key Test Cases | Expected Outcome |
|----------|-----------|----------------|------------------|
| `intervalSquared` | Unit | 1) Timelike separation: `(0,0,0,0)` → `(1,0,0,0)` ⇒ `-c²`<br>2) Lightlike: `(0,0,0,0)` → `(1,c,0,0)` ⇒ `0`<br>3) Spacelike: `(0,0,0,0)` → `(0,1,0,0)` ⇒ `+1` | Exact match (within double precision) |
| `properTime` | Unit | For timelike pair: should equal `√(-intervalSquared)/c`; for lightlike/spacelike return 0 or throw | 0 or appropriate error |
| `spatialDistance` | Unit | Between `(0,0,0,0)` and `(0,3,4,0)` ⇒ `5.0` | Exact |
| `spatialLength` | Unit | Origin → `(0,0,0,0)` ⇒ `0`; non‑zero value | Correct Euclidean norm |
| `theta`, `phi` | Unit | Test points on axes: `(0,0,1)` gives θ=0, φ=0; `(0,1,0)` gives θ=π/2, φ=π/2 | Expected spherical coordinates |
| `isTimelike/Lightlike/Spacelike` | Unit | Use same inputs as `intervalSquared` | Boolean correctly indicates sign |
| `lorentzTransform` | Unit | Boost along x with β=0.6: `(t,x)` → `(γ(t-βx), γ(x-βt), y, z)` | Compute analytically and compare |
| `operator+/-/*` | Unit | Simple vector arithmetic | Correct component‑wise operations |
| `operator==/!=` | Unit | Same vs. different events | True/false as expected |
| `operator[]` | Unit | Read/write components | Index maps correctly to t,x,y,z |

**Integration**: Use `Event4D` within `MetricTensor::interval` and verify consistency.

#### 4.1.2 MetricTensor

| Function | Test Type | Key Test Cases |
|----------|-----------|----------------|
| Default constructor | Unit | Should produce Minkowski metric (diag(-c²,1,1,1)) |
| `schwarzschild(mass, r, θ, φ)` | Unit | For large r → metric approaches Minkowski; at r=2GM/c², g_tt=0; manually compute a few components |
| `kerr(mass, a, r, θ)` | Unit | At equatorial plane, check g_tφ term; for a=0, must match Schwarzschild |
| `evaluate(event)` | Unit | Create metric with known components, verify `evaluate` returns correct array |
| `setToMinkowski/Zero` | Unit | Check all components after call |
| `set(mu, nu, val)` | Unit | Set component, read back |
| `interval(event1, event2)` | Integration | Compare with `Event4D::intervalSquared` when metric is Minkowski |
| `properTime` | Unit | For Minkowski, should match `Event4D::properTime` |
| `isLorentzian` | Unit | Minkowski: true; random symmetric matrix: false |
| `determinant` | Unit | Minkowski: -c²; Schwarzschild (external): -r⁴ sin²θ |
| `inverse` | Unit | Compute `metric * inverse`, should be identity |
| `lowerIndex/raiseIndex` | Unit | Apply to a known vector, verify index lowering/raising rules |
| Static constants | Unit | `C`, `C2`, `G` equal expected physical constants |

#### 4.1.3 LightCone

| Function | Test Type | Key Test Cases |
|----------|-----------|----------------|
| `computeFuture/Past` | Integration | For event at origin in Minkowski, set radius=1, check that all generated points satisfy Δs² ≤ 0 (future) or ≥ 0 (past) |
| `isCausal` | Unit | Test pairs: timelike ⇒ true, spacelike ⇒ false; lightlike ⇒ true |

### 4.6 Rendering Module

Rendering is inherently visual; we test via:

- **Initialization tests**: Call `initialize()` and check that no OpenGL error occurs, and `isInitialized()` returns true.
- **Smoke tests**: Render a known metric to an off‑screen framebuffer, capture the image (using `glReadPixels`), and compare against a reference image with tolerance (structural similarity index > 0.95).
- **Parameter tests**: After changing `setGridResolution` or `setCurvatureScale`, verify that the number of vertices in the generated mesh changes accordingly (via inspection of internal geometry arrays).

## 5. UI Automation Plan

Because QuantumVerse’s UI is built with Dear ImGui, direct widget access from test code is possible via `ImGui::GetCurrentContext()`. The plan:

1. **Headless rendering**: Use a headless OpenGL context (e.g., OSMesa or EGL) to run the app without a physical display.
2. **Scripted interaction**: Inject GLFW input events (`glfwSetKey`, `glfwSetCursorPos`) and call `UI4D_ImGui::render()` each frame.
3. **Visual validation**: Capture framebuffer content using `glReadPixels` and compare to baseline images stored in `tests/baseline_ui/`. Tolerance set to 2% pixel difference.
4. **Property checks**: Query ImGui internal state (e.g., `ImGuiWindow::Active`, `ImGui::GetIO().WantCaptureMouse`) to verify UI logic.

Test suites for each panel:

- **4D View**: Rotate camera, verify that curvature grid shader is active and a non‑black image is produced.
- **Slice Views**: Change slice offset, check that the displayed 4D coordinate label updates.
- **Planck Microscope**: Zoom scale changes, verify that the LOD level text matches.
- **Discovery Panel**: After running `detectAnomaly`, check that the anomaly table contains a new entry.

## 6. Performance & Stress Testing

### 6.1 Benchmarks (to be added to `tests/test_performance_baseline.cpp`)

| Target Function | Metric | Threshold |
|-----------------|--------|-----------|
| `Event4D::intervalSquared` | 100,000 calls | < 10 ms |
| `MetricTensor::schwarzschild` creation | 1,000 creations | < 50 ms |
| `GeodesicIntegrator::integrate` (timelike, 1000 steps) | Single integration | < 200 ms |
| `CausalSetEngine::grow(1000)` | 1000 elements | < 500 ms |
| `CurvatureRenderer::render` (1024×768) | Per frame | < 16 ms |

### 6.2 Stress Scenarios

- Grow causal set to 1,000,000 elements and measure memory/timing.
- Integrate 10,000 geodesics simultaneously (simulate spray of particles).
- Keep 100 anomaly alerts active and render FloatingPanels.
- Run all 10 discovery instruments on a complex metric and check for deadlocks.

### 6.3 Memory Leak Checks

- Run the entire test suite under AddressSanitizer (`build_asan` configuration) and Valgrind on Linux.
- Run long‑duration UI session (10,000 frames) and monitor `PerformanceMonitor::memoryUsedMB` growth (should stay below 2 GB baseline + 10% after 30 minutes).

## 8. Gap Analysis and Coverage Tracking

We will maintain a matrix linking each API function to one or more tests. Using `gcov/lcov` we will aim for **>90% line coverage** for all non‑UI modules, and >80% for UI modules (including ImGui rendering code).

Current status (51 tests):  
- Core physics: good  
- Quantum gravity: partial (only CDT, SpinFoam, GFT, CausalSet have basic tests)  
- Discovery: limited  
- UI: 0 automated UI tests  
- VR, ML: minimal  

**Missing tests to be developed** (estimated ~120 new test cases) are detailed in section 4 above.

## 10. Success Criteria

- All functions in the API Reference are covered by at least one automated test.
- Overall code coverage > 90% (non‑UI) and > 80% (UI).
- Zero critical or high‑severity bugs found during final acceptance runs.
- Performance benchmarks meet the thresholds listed in section 6.
- The system passes cross‑platform CI on Windows, Linux, and macOS.

## 11. Risk Mitigation

| Risk | Mitigation |
|------|------------|
| UI tests flaky due to rendering differences | Use structural similarity index (SSIM) with a generous tolerance (0.98); periodically update baselines |
| Headless OpenGL may not support all features | Provide a fallback using a virtual framebuffer (OSMesa) |
| Time‑consuming performance benchmarks | Run benchmarks only on PRs to main, or nightly |
| Missing ML models for ONNX tests | Include a tiny dummy model (2‑layer NN) in test data |

This plan ensures that QuantumVerse can be confidently declared “fully functional” and production‑ready.
